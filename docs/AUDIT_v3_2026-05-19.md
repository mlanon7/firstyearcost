# FirstYearCost — Deep Audit v3

**Date:** 2026-05-19
**Scope:** full project — content, math, rates, calculators, UI, SEO, data, monetization gaps, pre-deploy readiness
**Auditor:** Claude, with two parallel research agents (competitor SERP + 2026 rate verification)
**Method:** code review of every page and library file; CSV row inspection; npm gates (validate-data + tests + type-check + build); browser preview verification of state-page fix; live primary-source verification of every cited 2026 dollar figure (IRS / CMS / Federal Register / state PFML portals); SERP review of 9 high-intent queries

---

## Executive verdict

**This site is closer to launch than its previous audits suggest, but it has factual errors that must be fixed before traffic is pushed.** The architecture, math, palette, and UX polish are all in good shape. CDCTC + FSA + ACA OOP values are correct for 2026. CSV validation, 53/53 unit tests, and `tsc --noEmit` all pass.

**The trust-critical gaps left are factual, not structural:**
- **All 14 state PFML rows in `state_leave.csv` are stale.** Maximum weekly benefit caps moved on Jan 1, 2026 in California, Connecticut, Massachusetts, Minnesota, New Jersey, New York, Oregon, Rhode Island, Washington, DC, and Maine. Rhode Island moved from 7 to 8 paid weeks. **Maryland's program was delayed to January 2028 — the site currently shows it launching July 2026.** This is the single biggest accuracy hit.
- **Federal income-tax brackets in `subsidy.ts` are 2025 values, not 2026.** Affects FSA-savings estimates by up to a couple hundred dollars per scenario.
- **Production build fails to prerender `/icon`** on Windows (`@vercel/og` URL-decoding bug). Will likely succeed on Linux/Vercel but should be replaced with a static asset.
- **Privacy, FAQ, and methodology copy** still claim "no email signup" and "we don't account for FSAs/tax" — both false after v3 (EmailCapture exists; SubsidyCalculator exists).

**Also significant (from competitor research):** Two near-empty SERPs that the site is uniquely positioned to capture in the next 60–90 days — the **OBBBA $7,500 DCFSA + 50% CDCTC combo** (no parenting calc has updated) and **Trump Accounts** (launching July 5, 2026 with $1,000 federal seed per baby). Babylist already started moving on the second. Both are calculator-ready, fit the existing data layer, and align with where Google's Helpful Content updates have left the SERP unusually thin of credible competitors.

### P0-3. Federal income tax brackets in `subsidy.ts` are 2025 values

`lib/subsidy.ts:65-74` uses 2025 thresholds (single-filer: $11,925 / $48,475 / $103,350 / $197,300 / $250,525 / $626,350). The 2026 IRS Rev. Proc. 2025-32 (issued Oct 9 2025) updated to:

| Bracket | 2026 single | 2026 MFJ |
|---|---|---|
| 10% top | $12,400 | $24,800 |
| 12% top | $50,400 | $100,800 |
| 22% top | $105,700 | $211,400 |
| 24% top | $201,775 | $403,550 |
| 32% top | $256,225 | $512,450 |
| 35% top | $640,600 | **$768,700** ← *not* 2× single |
| 37% | above $640,600 | above $768,700 |

**Also:** `marginalBracketProxy` uses `const t = filing === 'mfj' ? 2 : 1;` and scales single thresholds × 2 for MFJ. This is wrong for the 32% and 35% bands (MFJ 35% top is $768,700, not 2 × $640,600 = $1,281,200). Replace the scale trick with two explicit bracket arrays.

**Fix:** rewrite `marginalBracketProxy` with proper 2026 single + MFJ arrays. HOH and MFS will need their own arrays too (MFS thresholds are roughly half MFJ; HOH thresholds differ in the lower brackets).

Source: [IRS Rev. Proc. 2025-32](https://www.irs.gov/pub/irs-drop/rp-25-32.pdf), [IRS IR-2025-103](https://www.irs.gov/newsroom/irs-releases-tax-inflation-adjustments-for-tax-year-2026-including-amendments-from-the-one-big-beautiful-bill).

3. Test the build directly on Vercel (push and verify) — acceptable if local-build pass is not required for CI.

## P1 — Should fix before launch but won't block

### P1-1. CSP / icon route audit recap

The CSP config is provider-aware (good) but no `NEXT_PUBLIC_*` env vars are set at build, so production runs with the minimum CSP. When you wire AdSense / GA4 / Plausible / Clarity / newsletter, the CSP auto-expands. Verify each provider's hosts after wiring; some (e.g., Mediavine, Sortable) need additional `script-src` entries not currently allowlisted.

### P1-2. Marginal bracket proxy ignores Saver's Credit, ACTC, and state taxes

`subsidy.ts:107` computes `fsaSavings = fsaUsed × (bracket + 7.65%)`. This is reasonable for federal-only, but state-tax savings (often another 3–10%) are not included. Worth a note in the SubsidyCalculator copy: "State tax savings on FSA dollars not modeled — add ~3–10% in states with income tax."

### P1-3. `calculate()` monthly distribution issues (carried forward)

Auditor flagged earlier:
- **Feeding** distributed flat across 12 months. Formula consumption actually peaks months 1–6 and tapers as solids start. Worth modeling a curve.
- **Medical** put 80% in month 1, 20% spread over 11 months. The full birth bill arrives in month 1 (or month 0 / pre-birth in some plans), not 80/20. Acceptable for a planning chart but the curve doesn't match reality.

### P1-4. State_leave.csv has no `source_id` column

The other row-level-source CSVs (`birth_oop_ranges`, `newborn_medical_oop`, `birth_billed_anchors`) have a `source_id` column referencing `source_registry.csv`. State leave only has `source_url` (the state's portal URL). The validator can't cross-check it against the registry.

**Fix:** Add a `source_id` column with values like `state-pfml-ca` / `state-pfml-co` etc., then add corresponding registry entries. Or accept the URL-only pattern and document it in the validator.

### P1-5. `npm run lint` is not configured

`next lint` triggers the first-time ESLint setup prompt every run. Until configured:
- CI runs that include `npm run lint` will hang/fail in non-interactive mode.
- Real ESLint warnings (unused vars, dead code, missing alt text, etc.) aren't caught.

**Fix:** Run `npx next lint --strict` once locally to generate `.eslintrc.json`, commit it, then `npm run lint` runs unattended.

### P1-6. `ChildcareCalculator` (`childcare-calculator/page.tsx`) doesn't disable months slider when care type would be inappropriate

The MainCalculator now correctly auto-zeros `childcareMonths` when plan is `none`/`family`. The standalone `ChildcareCalculator` has a slightly different state model and doesn't replicate this UX. Verify the behavior matches.

### P1-7. `/icon` route fallback recommended even if Vercel build passes

Drop `app/icon.tsx` and use static `app/icon.png` / `app/apple-icon.png`. Static binary is faster, smaller, and avoids the entire edge-runtime + @vercel/og dependency path.

### P1-8. `defaultInputs.parentalLeaveUnpaidMonths = 2` but the calculator doesn't use it for any number

Only used in the `notes` array on line 409 of `calculator.ts`. A user who tweaks the slider sees no number change. Two options:
- Remove the input from the UI entirely (per audit's "make every visible input affect the result").
- Or build the lost-income side-calculator the field implies — pair it with a household-income input.

### P1-9. Cross-page link consistency

- Footer links to "Daycare vs. nanny" (`/daycare-vs-nanny-cost`) and "C-section vs. vaginal cost" — good.
- Header has 9 nav items at desktop; per the GPT audit, replace "Home" / "Leave" / "Tax credits" with cleaner labels and add active-state highlighting.

### P1-10. No visible breadcrumbs

`BreadcrumbsJsonLd` provides structured data on every non-home page, but there's no rendered breadcrumb UI for users. On deep state pages (e.g., `/state-childcare-costs/california`) this would aid orientation and SEO.

### P1-11. No shareable calculator-state URLs

Calculator inputs are React state only. No way to share a scenario via URL. The audit recommended this as a real backlink driver — couples share a number with each other, Reddit/Facebook shares preserve the scenario. Implementation: serialize 14 inputs into URL search params, hydrate on mount.
